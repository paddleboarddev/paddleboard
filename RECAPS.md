### Sandboxed kernel — fix notebook cells wedged in "Running..." when the kernel never starts
- Symptom: after a Mac crash + restart, picking "Sandboxed Python" and running a cell showed `Running...` indefinitely. Diagnosed live: the Podman machine was mid-restart (`machinestate: Currently starting`), the `localhost/paddleboard-repl-python:1` image hadn't survived (just `python:3.12-slim` base in the cache), and the kernel-launch task failed before `Kernel::RunningKernel` was reached. End-to-end smoke test of the kernel-launch path (manual `podman build` of the Containerfile, `podman run` with bind-mounted connection.json + 5 ports + `/work`, `podman exec cat /tmp/connection.json`, host `nc -z` against each published port) all passed — so the runtime mechanics are fine; the wedge is a UX bug.
- Root cause in `crates/repl/src/notebook/notebook_ui.rs`: `execute_cell` flipped `is_executing=true` on the cell unconditionally, then only sent the `ExecuteRequest` `if let Kernel::RunningKernel(kernel) = …`. When the kernel was `StartingKernel` or `ErroredLaunch`, the request was silently dropped but nothing ever calls `finish_execution()`, so the cell renders the "Running..." chip forever. Compounding: `kernel_errored` flipped `self.kernel` to `ErroredLaunch` without touching any already-in-flight cells. The inline-REPL path in `session.rs::execute` doesn't have either bug — it assigns a per-block `ExecutionStatus` derived from the kernel state, and its `kernel_errored` walks `self.blocks` setting status to `KernelErrored`.
- Fix (notebook path only): (1) `execute_cell` now early-returns with a `log::warn!` if `!matches!(self.kernel, Kernel::RunningKernel(_))`, before `start_execution()` or `execution_requests.insert(...)`. (2) `kernel_errored` now `drain()`s `execution_requests` and calls `cell.finish_execution()` on each pending cell, so cells in-flight at the moment of kernel death also unstick. Multi-line `// PaddleBoard:` comments at both sites explain the trap so the next person doesn't re-introduce it.
- Verified: `cargo check -p repl` clean (7.6s; 0.86s incremental). Did NOT smoke-test end-to-end again — the runtime path was already proven by the manual `podman run` earlier in the session.
- Preserved: inline-REPL path in `session.rs` (already correct); the kernel-status pill at the top of the notebook (it already surfaces Starting/Errored/Shutdown, which is the user-visible signal once cells no longer hijack attention).
- Follow-ups: (1) `./script/clippy -p repl` is currently red on a pre-existing hit in `podman_kernel.rs:510` — `Drop::drop` calls `std::process::Command::output()` which the project's `disallowed-methods` lint forbids. Drop is sync so `smol::process::Command` doesn't fit; the right fix is an `#[allow(clippy::disallowed_methods)]` with a comment explaining why sync is required in Drop. Not in scope for this commit. (2) Better recovery for the `StartingKernel` case: queue execute requests and replay them when the kernel reaches `RunningKernel`, rather than silently dropping. Would require an `execution_requests` deferred-send queue and care around msg_id/cell_id lifetime. Out of scope here — the simple early-return is enough to stop the wedge.
- Files changed: `crates/repl/src/notebook/notebook_ui.rs` (two edits, ~30 lines net).

### Sandboxed kernel — fix `podman build` shipping the entire repo as context
- Smoke test finally surfaced the *actual* reason the kernel wouldn't start. PaddleBoard log: `ERROR [repl::notebook::notebook_ui] Kernel failed to start: podman build for localhost/paddleboard-repl-python:1 failed: Error: invalid symlink "/var/tmp/libpod_builder.../webrtc-sys" -> "../../../../../../.cargo/git/checkouts/livekit-rust-sdks-.../webrtc-sys"`.
- Diagnosis: my invocation was `podman build --tag <tag> --file - .`. The trailing `.` is the build context, and on macOS Podman tarballs the *entire context directory* and ships it to the Podman machine VM. PaddleBoard's cwd is the repo root, which contains `target/debug/build/...` from prior cargo builds. That tree has a symlink at `target/debug/build/webrtc-sys-.../out/cxxbridge/crate/webrtc-sys` pointing to `~/.cargo/git/checkouts/livekit-rust-sdks-.../webrtc-sys` — outside the context — and podman correctly refuses to follow it.
- Fix: switched to `podman build --tag <tag> -`. The single `-` positional arg tells Podman "read the Containerfile from stdin, no build context." Our Containerfile is self-contained (`FROM python:3.12-slim` + `pip install ipykernel`), so we never needed any files from the host repo as build inputs.
- Added a multi-line comment at the call site explaining the trap so the next person doesn't re-introduce a context path. Files changed: `crates/repl/src/kernels/podman_kernel.rs` (one-line fix + comment).
- Verified: `cargo check -p repl` clean (9.49s); rebuilt and relaunched paddleboard. Awaiting next smoke-test attempt from the user — `podman build` should now actually run (first launch pulls `python:3.12-slim` ~50MB so still 1-3 minutes).

### Sandbox prerequisites detection (PR-A of two)
- New crate `paddleboard_sandbox_prereqs` with `check() -> SandboxStatus`. Probes `podman --version`, `podman info --format json`, and parses the JSON for `host.ociRuntimes.runsc`. Each probe is timeout-bounded (2s) so a stuck `podman machine` cannot stall startup.
- Three variants per dimension: `PodmanStatus` (Missing / InstalledNotRunning / Ready) and `GvisorStatus` (Available / NotConfigured / NotApplicable / Unknown). On macOS the InstalledNotRunning state is the common case when `podman machine` is stopped.

### Fixed shell-interpolation injection pattern in merge_upstream_zed.yml
- `cargo xtask check-workflows` (exposed after PR #21) flagged 8 instances of `${{ steps.*.outputs.* }}` being interpolated directly into shell `run:` blocks across 5 steps. That's GitHub's documented script-injection pattern — values from `${{ }}` expressions are substituted *before* the shell parses the script, so a value containing shell metacharacters would be executed.
- In this specific workflow the values come from `git rev-parse` / `git merge-base` / `date -u` and are safe in practice, but the validator is right that the pattern is dangerous and worth fixing on its own merits.
- Moved each output into a step-level `env:` block, switched `run:` blocks to read `$VAR` instead of `${{ }}`. `if:` conditions on step outputs stay as-is (those are GitHub Actions expressions evaluated before any shell runs — safe and correct).
- Validator now exits 0 against the workflow.
