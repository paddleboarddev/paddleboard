### Scaffold upstream-drift dashboard
- Followup to the upsell/Zeta removal: a recurring CI check that measures how far PaddleBoard's upstream-shaped crates have drifted from `zed-industries/zed:main`. Complements (doesn't replace) the existing weekly `merge_upstream_zed.yml` action — surfaces drift trend between merges so the next merge isn't a surprise.
- **New `script/check-upstream-drift`** (~120 LOC bash). For each crate in a curated `CORE_CRATES` list (agent, agent_ui, buffer_diff, client, collab, collab_ui, editor, gpui, language, language_core, lsp, multi_buffer, project, rope, settings_content, text, workspace), runs `git diff --shortstat zed/main -- crates/<name>` and pairs that with a count of `// PaddleBoard:` divergence markers in the same path. Markdown-formatted table to stdout. Locally runnable as `./script/check-upstream-drift` (with `--no-fetch` to skip the remote fetch when iterating). One gotcha encountered during smoke-testing: `set -o pipefail` made the `git grep -c ... | awk` pipeline abort on crates with zero markers (`git grep -c` exits 1 when no matches); wrapped the pipeline in a subshell that disables pipefail locally so empty matches count as zero.
- **New `.github/workflows/upstream-drift.yml`**. Three triggers: every PR (posts a sticky comment), Sundays 14:00 UTC (writes the report to `$GITHUB_STEP_SUMMARY`), and push to the dashboard's own branches (`chore/upstream-drift-dashboard*` / `fix/upstream-drift-*`). The `push` trigger was discovered necessary during smoke-testing — GitHub's `pull_request` trigger only registers workflows that already exist on the default branch, so a brand-new workflow added in a PR doesn't run on its own PR. The narrow `push` trigger fires on the dashboard's own branch so the first run happens before merge; after merge, `pull_request` takes over and the `push` becomes a no-op for typical work. Sticky comment uses an HTML-comment marker (`<!-- upstream-drift-report -->`) so subsequent pushes update the existing comment via `gh api -X PATCH` rather than spawning new ones.
- Curated crate list excludes `paddleboard_*` (net-new, no upstream baseline), `grammars/` (we own the dir layout for the language bake-ins), and the long tail of smaller-blast-radius crates. Easy to add to the list when a specific incident motivates instrumenting another path.
- Current snapshot (HEAD vs `zed/main`): 388 files diff'd across the 17 tracked crates, +84,938 / -85,012 LOC, 9 `// PaddleBoard:` tags. The big diff numbers reflect both PaddleBoard divergence AND upstream changes since the last common ancestor (`3a5dc8ef6a`); after a weekly merge brings upstream in, these collapse back to just PaddleBoard divergence. Interpretation guide is in the workflow's header comment.
- Verified: `./script/check-upstream-drift --no-fetch` runs locally in under a second; the first CI run via the push trigger succeeded at <https://github.com/jasonsmithio/paddleboard/actions/runs/25975486647> with identical numbers and full table written to the workflow summary. Sticky-comment plumbing is NOT exercised by the push run (gated on `event_name == 'pull_request'`) — will be tested on the first PR after this merges.
- Preserved: the existing `merge_upstream_zed.yml` workflow, `FORK_HYGIENE.md` doc, and the `// PaddleBoard:` marker convention. The drift script reads those markers; it doesn't generate them.
- Follow-ups: (1) once the dashboard's been running for a few weeks, tune thresholds — e.g. fail the check if drift in a single crate jumps by >5,000 LOC week-over-week, indicating an unintentional fork. (2) Consider tracking drift against upstream release *tags* (more stable than `main`) once Zed cuts a release. (3) If the `gh api -X PATCH --field body=@file` invocation behaves badly on the first real PR, fall back to creating a new comment each time.

### Sandboxed kernel — fix notebook cells wedged in "Running..." when the kernel never starts
- Symptom: after a Mac crash + restart, picking "Sandboxed Python" and running a cell showed `Running...` indefinitely. - Root cause in `crates/repl/src/notebook/notebook_ui.rs`: `execute_cell` flipped `is_executing=true` on the cell unconditionally, then only sent the `ExecuteRequest` `if let Kernel::RunningKernel(kernel) = …`. Compounding: `kernel_errored` flipped `self.kernel` to `ErroredLaunch` without touching any already-in-flight cells. The inline-REPL path in `session.rs::execute` doesn't have either bug — it assigns a per-block `ExecutionStatus` derived from the kernel state, and its `kernel_errored` walks `self.blocks` setting status to `KernelErrored`.

(2) `kernel_errored` now `drain()`s `execution_requests` and calls `cell.finish_execution()` on each pending cell, so cells in-flight at the moment of kernel death also unstick. Multi-line `// PaddleBoard:` comments at both sites explain the trap so the next person doesn't re-introduce it.

### Sandboxed kernel — fix `podman build` shipping the entire repo as context
- Smoke test finally surfaced the *actual* reason the kernel wouldn't start. PaddleBoard log: `ERROR [repl::notebook::notebook_ui] Kernel failed to start: podman build for localhost/paddleboard-repl-python:1 failed: Error: invalid symlink "/var/tmp/libpod_builder.../webrtc-sys" -> "../../../../../../.cargo/git/checkouts/livekit-rust-sdks-.../webrtc-sys"`.
- Diagnosis: my invocation was `podman build --tag <tag> --file - .`. The trailing `.` is the build context, and on macOS Podman tarballs the *entire context directory* and ships it to the Podman machine VM. PaddleBoard's cwd is the repo root, which contains `target/debug/build/...` from prior cargo builds. - Fix: switched to `podman build --tag <tag> -`. The single `-` positional arg tells Podman "read the Containerfile from stdin, no build context." Our Containerfile is self-contained (`FROM python:3.12-slim` + `pip install ipykernel`), so we never needed any files from the host repo as build inputs.
- Added a multi-line comment at the call site explaining the trap so the next person doesn't re-introduce a context path. Files changed: `crates/repl/src/kernels/podman_kernel.rs` (one-line fix + comment).

### Sandbox prerequisites detection (PR-A of two)

### Fixed shell-interpolation injection pattern in merge_upstream_zed.yml
- `cargo xtask check-workflows` (exposed after PR #21) flagged 8 instances of `${{ steps.*.outputs.* }}` being interpolated directly into shell `run:` blocks across 5 steps. That's GitHub's documented script-injection pattern — values from `${{ }}` expressions are substituted *before* the shell parses the script, so a value containing shell metacharacters would be executed.
- In this specific workflow the values come from `git rev-parse` / `git merge-base` / `date -u` and are safe in practice, but the validator is right that the pattern is dangerous and worth fixing on its own merits.
- Moved each output into a step-level `env:` block, switched `run:` blocks to read `$VAR` instead of `${{ }}`. `if:` conditions on step outputs stay as-is (those are GitHub Actions expressions evaluated before any shell runs — safe and correct).
- Validator now exits 0 against the workflow.
